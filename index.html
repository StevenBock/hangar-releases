01The Hangar · No agents required

One roof over your whole stack.

Even with zero agents running, Hangar earns its keep. Declare your project's commands once in hangar.yml and every dev server, watcher, and one-off tool gets a home — started in dependency order when you open the project, restartable from the sidebar, two keystrokes from anywhere. No more terminal-tab archaeology.

  • Declare once, run forever — "how do I run this repo?" becomes open the project; on-demand tools start their own dependency chain when triggered
  • Terminals that keep up — scrollback, search, and copy that don't flinch, even when the build output gets loud
  • Log files on the shelf — register a file once and it's tailed, filtered, and queryable right next to the process that writes it
  • Squawks — quick per-project notes with attachments, so the "remember to fix this" sticky lives with the project, not on your monitor bezel
02The Fleet

Run a crew of agents, not a pile of tabs.

Every agent is a first-class process with a real terminal, live status, and a place in the stack. Spawn Claude Code next to Codex next to OpenCode and watch them work — side by side with your dev servers and test watchers.

  • Live agent status — see who's executing, idle, or blocked waiting on you
  • Agents talk to each other — spawn siblings, hand off tasks, message across terminals
  • Sessions survive restarts — resume an agent right where it left off
03The Tool Deck · MCP

Your agents get hands.

Hangar ships a built-in MCP server that exposes the entire workspace as deterministic tools — 80+ of them, connected the moment an agent spawns. No config files, no glue scripts: an agent starts already knowing how to read the stack, query your telemetry, and coordinate with the other agents.

  • Read the room — list the stack, read any terminal's output, check the test watcher instead of re-running the suite
  • Coordinate agents — spawn siblings, message them mid-task, see who's idle, executing, or blocked
  • Operate the workspace — SQL over telemetry and logs, worktree lanes, schedules, the logbook wiki
  • Leave the place better — agents log what moved, update project memory, even author themes

04The Tower

⌘K clears everything for takeoff.

One keyboard-driven palette runs the whole show: spawn an agent, start the dev stack, jump to any surface, flip the theme. Your hands never leave the keys — every view in the app is two keystrokes away.

  • New Agent: Claude Code / Codex / OpenCode — a fresh agent in one keystroke
  • Process controls — start, restart, and jump to anything in the stack
  • Keyboard everything — themes, sidebar, quick switch, even cat mode: if it's in Hangar, it's in the palette
05The Flight Deck

Telemetry your agents can read too.

A built-in OTLP receiver turns your app's traces, logs, and metrics into a live diagnostic view: error groups, latency heatmaps, slow operations, and SQL pathologies — spotted automatically. And because it's all exposed over MCP, your agents diagnose from the same data you do.

  • Pathology detection — N+1 query runs and SQL hotspots surfaced, scored, ranked
  • Error groups with symptoms — related failures across services, folded into one story
  • Point an agent at a trace — "why is checkout slow?" is now a one-liner
06The Black Box

405,000 log lines. One grid.

Register any log file and Hangar parses it — timestamps, levels, and structured fields become real columns. Filter by level, search messages, scrub the timeline histogram, or drop into a full SQL console when a filter bar isn't enough.

  • Built for volume — hundreds of thousands of rows, instant scrubbing
  • SQL console + patterns — real queries over parsed fields, recurring shapes surfaced
  • Agents query it over MCP — "find the errors before the crash" answers itself
07Pre-Flight Checks

Diffs that read like instruments.

A git surface built for reviewing agent output: anchor-aligned split diffs with full syntax highlighting, commit history, and an Ask Agent button on every file — so "explain this hunk" or "fix this" goes straight to a terminal with context already loaded.

  • Smart diffs — two-tone, inline-highlighted, aligned so your eyes don't hunt
  • History included — every commit a click away, rendered with the same care
  • Review, then ask — hand any diff to an agent without copy-pasting a thing
08The Runway · Blueprints × Lanes

Plan it on a blueprint. Build it in a lane.

Big features don't belong in your working checkout. Draft the plan with an agent as a blueprint — a living document with acceptance criteria and interactive prototype pages — then hit Build this: the agent gets a lane, an isolated git worktree on its own branch, and works the plan while your files stay yours.

  • Blueprints are contracts — a human-readable doc, machine-checkable acceptance criteria, and prototypes you can click before a line of code exists
  • Lanes are clean rooms — every build gets its own worktree and branch; run three agents on three features with zero collisions
  • One-click handoff — Build spawns the agent, seeds it the plan, and criteria tick as they hold
  • Land when green — sync with main, review the diff in the git view, land the lane with one command
